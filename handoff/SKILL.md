# Handoff — 会话交接文档生成

当前会话的上下文即将耗尽，需要生成一份交接文档，让下一个会话的 Agent 能快速理解现状并直接继续工作。

## 前置步骤：恢复上下文

在写交接文档前，确保你掌握了当前会话的完整工作内容。按需执行以下检查：

1. **恢复被 compact 的上下文** — 如果当前会话经历过 compact，读取会话 transcript 恢复丢失的信息。transcript 路径格式：`~/.claude/projects/{project-path-with-dashes}/{session-id}.jsonl`（JSONL 格式，每行一个 JSON 对象，包含 user/assistant 消息）。优先读最后 200 行获取最近上下文，再按需向前追溯。
2. **捕获会话内 task 状态** — 如果当前会话使用了 TaskCreate/TaskList，调用 `TaskList` 获取所有任务状态。这些 task 是会话级的，下一个会话无法访问，必须在交接文档中完整记录。
3. **回顾本会话涉及的 plan/spec 文件** — 你在本次会话中已经读过或参与修改的 plan 文件（GSD 的 CONTEXT.md/PLAN.md、superpowers 的 specs/plans、Claude Code 内置 plan 等），在交接文档中标注其路径和当前执行状态（未开始/进行中/已完成）。不需要扫描目录去发现新文件——你在会话中没接触过的 plan 对交接没有意义。
4. **检查 git 状态**：
   - `git status` — 未提交的改动和当前分支
   - `git diff --stat` — 改动文件和行数概览
   - `git log --oneline -10` — 最近提交记录

## 输出

在项目根目录生成 `./{yymmdd}-handoff.md`（用当天实际日期，如 `260329-handoff.md`）。

如果同名文件已存在（同一天的多次交接），追加序号：`260329-handoff-2.md`、`260329-handoff-3.md`。

## 文档结构

按以下 7 个章节整理交接内容。每个章节必须包含具体、可执行的信息，杜绝空话。

### 1. 当前任务目标
说明当前要解决的问题、预期产出和完成标准。
- 如果有多个并行任务，按优先级列出
- 引用具体的需求编号、phase 名称、issue 编号等

### 2. 当前进展
说明目前已经完成了哪些分析、确认、修改、排查、讨论或产出。
- 用表格或列表清晰标注每项的完成状态（✅ 已完成 / ❌ 未完成 / 🚧 进行中）
- 列出已产出的文件路径
- 如果当前会话有 task list，将所有 task 及其状态完整列出

### 3. 关键上下文
包括但不限于：
- 重要背景信息
- 用户的明确要求和偏好
- 已知约束（技术限制、时间限制、依赖关系）
- 已做出的关键决定及其理由
- 重要假设（标注哪些已验证、哪些待确认）

### 4. 关键发现
列出目前最重要的结论、规律、异常点、根因判断、设计判断或值得注意的信息。
- 每条发现用一句话概括，必要时附上证据来源（文件路径、命令输出、日志行号）
- 明确标注"已验证"和"待确认"

### 5. 未完成事项
列出仍需要继续处理的内容，按优先级排序（P0 阻塞性 / P1 重要 / P2 可延后）。
- 每项说明当前卡在哪里、需要什么才能继续
- 如果某项已尝试过但失败，说明失败原因，避免下一位 Agent 重复尝试

### 6. 建议接手路径
告诉下一位 Agent：
- 应优先查看哪些文件、模块、数据、日志、命令、页面或线索
- 应先验证什么（确认环境、确认状态、确认前提条件）
- 推荐的下一步动作序列
- 如果项目根目录存在更早的 `*-handoff.md` 文件，提醒下一位 Agent 可参考历史交接获取更多背景

### 7. 风险与注意事项
- 哪些点容易误判或跑偏
- 哪些方向已经验证过且不建议重复尝试
- 哪些隐含依赖或副作用需要注意

## 写作原则

- **总长度控制在 150 行以内** — 交接文档本身也会占用下一个 Agent 的上下文，过长则适得其反
- **引用代替复述** — 如果信息已经存在于项目文件中（如 `07-CONTEXT.md` 的决策、`ROADMAP.md` 的进度），只写一句话概括 + 文件路径，不要把内容抄进来。下一个 Agent 可以自己去读原文件
- **只记录文件中读不到的信息** — 会话中的口头决定、用户偏好、调试过程中的排除项、尝试过但失败的方案——这些只存在于对话中，是交接文档最该保留的内容
- **受众是 AI Agent，不是人类用户** — 不需要客套话，直接给事实和行动指引
- **具体优先** — 引用文件路径、类名、函数名、命令、API 端点、变量名、决策编号
- **可执行优先** — 每条信息都应该帮助下一位 Agent 做出决策或采取行动
- **诚实标注不确定性** — 不确定的信息标注"待确认"，不要臆造

## 末尾

文档最后必须有一段 **"下一位 Agent 的第一步建议"**，给出 3-5 步具体操作指引，让接手 Agent 读完就能开始工作。

## 生成后：提示用户下一步

Handoff 文件生成后，根据使用场景提示用户：

**场景 A：上下文即将耗尽，需要切换会话**
> handoff 已生成。建议退出当前会话，开一个新的 claude 会话，然后告诉新 Agent：
> `读 {handoff文件名}，继续之前的工作`

**场景 B：保存当前进度，下次再继续**
> 进度已保存到 `{handoff文件名}`。下次打开 claude 时，告诉 Agent：
> `读 {handoff文件名}，继续之前的工作`

两种场景的区别只在于时机——A 是立即切换，B 是稍后恢复。handoff 文件的内容和格式完全一样。
